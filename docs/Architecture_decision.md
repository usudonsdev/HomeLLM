# アーキテクチャ意思決定記録 (architecture_decision_records.md)

## ADR-001: 実行基盤としての Kubernetes (k8s) の採用

### 【コンテキスト】
当初は Docker Compose によるシンプルな複数コンテナ運用を想定していた。しかし、「常時起動でレスポンスが求められる就活Webサイト」と、「不定期かつリソースを極限まで消費するゲーム動画解析システム」という、性質と負荷の異なるタスクを1台のゲーミングノートで同居させる必要が生じた。

### 【判定】
WindowsのWSL2上に軽量なk8sディストリビューション（k3s または MicroK8s）を構築し、マイクロサービスアーキテクチャへ移行する。

### 【根拠】
1. **厳密なリソース分離**: Docker Composeでは難しい、Namespace単位でのCPU/メモリの `Limits`（上限設定）を適用し、動画解析時の負荷からWebサイトの可用性を死守するため。
2. **ライフサイクル管理の最適化**: 動画解析を常時起動コンテナではなく、k8sの `Job` アブストラクトとして定義することで、処理終了時に自動でPod（リソース）を完全解放するため。
3. **学習価値の最大化**: マイクロサービスにおけるインフラ管理・サービス間通信の標準パターンを実システムで経験するため。

## ADR-002: Ollama のコンテナ外（Windowsネイティブ）配置

### 【コンテキスト】
AI推論を行う Ollama をk8sクラスターのPodとして内包すべきか、Windowsの生環境に残すべきかの検討。

### 【判定】
Ollama はk8s（WSL2）の内部にカプセル化せず、Windowsのネイティブ環境に配置したまま、各マイクロサービスからネットワーク経由で利用する。

### 【根拠】
- **コンテナ内GPU透過の複雑さ**: WSL2内のDocker/k8sからNVIDIA GPU（RTX）を安定して叩くには、NVIDIA Container Toolkitのレイヤーが挟まり、環境依存のバグやアップデート時の破壊リスクが跳ね上がる。
- **結合度の低下**: Ollamaをコンテナ外の「独立した共有AI推論API」として扱うことで、インフラ構成（マニフェスト）がシンプルになり、デスクトップ環境で作成したマニフェストをWindowsノートへ移植する際のポータビリティが100%保たれるため。

## ADR-003: Phase 1 から k8s 直進（Compose 先行しない）

### 【コンテキスト】
最終ゴールはマイクロサービス構成である。初期速度を優先して Docker Compose で先行実装し、後から k8s へ移行する案（A）と、最初から軽量 k8s 上に構築する案（B）を比較した。

### 【判定】
**案 B を採用する。** Phase 1 から Windows（WSL2）上の **k3s** に API / DB / 将来の Job を載せる。Compose は移行用ランタイムとしては採用しない（局所的な開発補助に限る）。Web フロントは ADR-005 により Raspberry Pi へ置く。

### 【根拠】
1. **ゲーム別 pipeline の分散が必然**: 動画解析はゲームごとに分割・分析フローが異なる（Valorant はラウンド間ロゴ検出、TFT は時間基準セグメント等）。単一ワーカーへ溶かすと後のサービス分割コストが大きい。
2. **Job 並列と隔離**: 分割後クリップ／セグメント単位で AI Job を走らせるため、Namespace・ResourceQuota・Job ライフサイクルを最初から使える形にしておく。
3. **移行イベントの回避**: 正しく分割した Compose からの移行でも 1〜2 週間規模の載せ替えが発生する。最終形が分散前提なら、最初から k3s に払う方が総コストが明確。

### 【補足: 重い推論と k3s の役割分担】
- ホスト Ollama の **`qwen3.5:9b`** はリソース消費が大きいが、ADR-002 により **k3s ResourceQuota の外側**で動く。
- LLM 負荷はアプリ側の同時推論制限・キューイング、および運用ルールで制御する。
- k3s 側は API/DB と動画 Job の隔離・並列度制御を担う。Web 常時配信は Pi（ADR-005）。
- Swarm は単機＋バッチ Job 要件に対して旨味が薄いため採用しない。

## ADR-004: 動画解析はゲーム別 pipeline として分割する

### 【コンテキスト】
複数タイトル（当面 Valorant、続いて Teamfight Tactics 等）のリプレイ／プレイ動画を解析し、得られた知見を `job-hunting` の「引き出し」へ蓄積したい。タイトルごとにカット点の取り方が根本的に異なる。

### 【判定】
`video-analysis` ドメイン内を **共通 ingest + ゲーム別 pipeline** に分割する。ゲーム固有ロジックを単一サービスに閉じ込めない。

### 【根拠】
- フロー差分が仕様レベルで大きいため、共有できるのは受付・保存・結果送信などの薄い層に限られる。
- 分割単位（ラウンド／時間セグメント）ごとに後続 AI Job を起動するため、pipeline を分けた方が Job 定義とスケール方針を独立させられる。

## ADR-005: Web は Raspberry Pi、重い処理は Windows

### 【コンテキスト】
エッジは常時起動の Raspberry Pi、計算は LLM 用 Windows ノートのみ。Windows に Web まで載せると、常時配信と重い推論・解析が同じ筐体で競合し、電気代・WOL 運用とも相性が悪い。

### 【判定】
- **Raspberry Pi**: Web フロントエンドのホスティング、Tailscale、WOL
- **Windows**: k3s 上の API / PostgreSQL / 動画 Job、およびホスト Ollama（`qwen3.5:9b`）
- Pi に DB・LLM・動画処理を載せない

### 【根拠】
1. Windows を重い処理に集中させ、常時負荷を Pi に寄せる
2. Web 可用性を Windows のスリープ／WOL から切り離せる（API が落ちている間の UX は別途設計）
3. 公開面を Pi の薄いフロントに限定し、個人データと GPU を Windows 側に閉じやすい

### 【補足】
API 停止中（Windows off）のフロント挙動（メンテ表示、WOL トリガ等）は実装時に決める。

## ADR-006: 公開 GitHub を前提とした秘密情報の扱い

### 【コンテキスト】
リポジトリを GitHub 公開で運用する。就活・経験データは究極の個人情報であり、自宅インフラの実識別子も攻撃面になりうる。

### 【判定】
- 実 IP / MAC / パスワード / 生の経験ログはリポジトリに含めない
- 設定はプレースホルダと `*.example` のみ公開し、実値は gitignored な private 領域またはシークレットマネージャへ
- 詳細ルールは `docs/security.md` を正とする

### 【根拠】
公開による学習・ポートフォリオ価値を残しつつ、個人情報と自宅ネットワークの露出を最小化するため。
