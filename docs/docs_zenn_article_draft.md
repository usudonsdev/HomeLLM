## はじめに
就活のエントリーシート（ES）や面接の反省、学生時代の泥臭い失敗談や感情のログ。これらはすべて**究極の個人情報の塊**です。ChatGPTなどのクラウドAIにこれらを生のまま突っ込むのはセキュリティ的に気が引ける…。

そこで、手元で眠っていた「強力なGPUを積んだゲーミングノートPC」を活用し、完全オフラインで動作する自分専用の就活データ解析サーバーを構築しました。
しかし、構築の過程で**「画面を閉じるとスリープする」「ゲーミングノート特有の電気代がやばい」**という現実的な壁にぶつかりました。

本記事では、この問題を**Raspberry Pi（WOL大砲）**と**Tailscale**、そして**Docker + Cursor**を組み合わせてスマートに解決した一連の構成と、ローカルLLMを「引き出し」として最適に機能させる設計思想について解説します。

## 全体システムアーキテクチャ
本システムの全体像は以下の通りです。

1. **開発**: デスクトップPCの Cursor を使い、GitHub (Private) 経由でコードを管理。
2. **本番環境**: Windows ゲーミングノート内の Docker コンテナ（PostgreSQL + FastAPI）。
3. **AI基盤**: Windows 生環境の Ollama（Port 11434）。
4. **インフラ**: Tailscale によるメッシュネットワーク。
5. **電源管理**: Raspberry Pi からの有線LANマジックパケット（WOL）。

## 1. ゲーミングノートの「電気代」と「スリープ」の壁を越える
Windowsのデフォルト設定では、ノートPCのディスプレイを畳むと強制的にスリープ状態になります。電源オプションから「カバーを閉じたときの動作: 何もしない」に変更することで、画面を閉じてもサーバーとして稼働させることが可能になります。

しかし、ゲーミングノートPCはアイドル時でも**30W〜50W**もの電力を消費します。24時間30日間つけっぱなしにすると、毎月1,000円前後の電気代が何もせずとも消えていく計算になります。

### 解決策: 使うときだけラズパイから遠隔起動（Wake-on-LAN）
常時起動している自宅の Raspberry Pi に `etherwake` を仕込み、Macやデスクトップからラズパイ経由で Windows の有線LANチップに向けてマジックパケットを撃ち出す構成にしました。

```bash
# ラズパイ側からWindowsの有線LANのMACアドレスを指定して叩き起こす
sudo apt install etherwake -y
sudo etherwake -i eth0 aa:bb:cc:dd:ee:ff
```
これで、電気代がかかる夜間や大学にいる間は Windows サーバーを完全にスリープさせておき、開発や分析を行いたい瞬間に遠隔で安全に叩き起こす仕組みが完成しました。

## 2. クラウドAIではダメだった理由と、ローカルRAGの設計思想
最初はクラウドLLMにあらゆる経験（授業、インターン、個人製作、部活、イベント）のログをすべて食わせていましたが、情報過多によってコンテキストの真ん中が無視される現象（Lost in the Middle）が発生したり、AIが綺麗に要約しようとするあまり、当時の「泥臭い試行錯誤のディテール」が消えてしまうという問題に直面しました。

そこで、データ自体は RDB（PostgreSQL）でカテゴリやタグごとに細かく小分け（構造化）して保存。
AIで引き出したいとき（例: 「トラブルシューティングの経験」）は、バックエンド側で関連する少数の生のログだけをフィルタリングして Ollama に渡す「ローカルRAG」の仕組みにしました。これによって、コンテキスト制限の厳しいローカルLLMでも、非常に解像度の高い自己分析と言語化が行えるようになりました。

## 3. デスクトップ設計 ➔ GitHub ➔ Windowsデプロイのスマートな開発フロー
開発の快適性を落とさないよう、以下のフローを構築しました。
1. デスクトップの Cursor を使い、リポジトリのルートにプロンプト・設計書の塊である `.cursorrules` を配置。
2. データベース（PostgreSQL）のコンテナ設定を `docker-compose.yml` に記述。
3. コード一式を GitHub の Private リポジトリに Push。
4. Windows ゲーミングノート側で `git pull` し、`docker-compose up -d` でバックグラウンド起動。

※バグを回避するため、Ollama 自体は Docker コンテナ内に入れず、Windowsのネイティブ環境で動いているものをコンテナ内からホストIP経由で叩く構成にしているのがポイントです。
